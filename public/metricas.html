Estimación para bases de datos multimedia en MongoDB

Lectura en vivo de tu colección personas: costo de consulta (explain), proyección de almacenamiento (collStats), estrategias de indexación y tamaño BSON. Abajo tienes la guía detallada de cómo hacer cada análisis, gráficos basados en tus datos y cómo reutilizar el mismo enfoque gráfico en otros estudios.

Cómo deben hacerse estos análisis (metodología)

Cada bloque resume el objetivo, los pasos reproducibles, las métricas que debes anotar y los errores habituales. En laboratorio se recomienda repetir mediciones tras cambiar índices o volumen de datos y guardar capturas de explain y collStats para comparar versiones.

1. Almacenamiento y collStats

Objetivo. Cuantificar cuánto ocupan los documentos frente a los índices y estimar crecimiento.

  1. Fija el contexto: base de datos (DB_NAME) y colección (aquí personas). En multimedia sueles repetir el proceso para metadatos y para GridFS (fs.files, fs.chunks).
  2. Ejecuta estadísticas de colección. En mongosh: db.personas.aggregate([{ $collStats: {} }]) o db.runCommand({ collStats: "personas" }). Esta página llama al mismo comando vía API.
  3. Anota count, size (bytes lógicos de datos sin contar índices), avgObjSize, storageSize (espacio asignado en disco, incluye fragmentación/padding) y totalIndexSize.
  4. Interpreta: si storageSize ≫ size, puede haber fragmentación o eliminaciones recientes; un compact (con cuidado en producción) o la compactación natural tras reescrituras puede cambiar el ratio.
  5. Para proyección, usa avgObjSize × documentos_nuevos como primera aproximación y suma el overhead de índices (ratio observado totalIndexSize / size o un porcentaje acordado en tu informe).
Documentación útil: manual MongoDB «collStats» — revisa definiciones exactas de cada campo según tu versión del servidor.
2. Tamaño BSON por documento

Objetivo. Entender la dispersión del peso de cada registro (campos opcionales, strings largos, arrays de referencias a multimedia).

  1. Obtén una muestra representativa: no solo los últimos insertados, salvo que tu caso de uso sea solo “recientes”. Para otro estudio puedes usar $sample en una agregación.
  2. Mide el BSON serializado. En Node se usa calculateObjectSize(doc) del paquete bson (lo que hace el servidor internamente es coherente con el tamaño en wire protocol).
  3. Compara el promedio muestral con avgObjSize de collStats: si difieren mucho, la muestra puede no ser representativa o hay documentos históricos con esquema distinto.
  4. En multimedia, separa conceptualmente metadatos (BSON pequeño) del binario almacenado en GridFS o fuera del documento; el tamaño del documento de metadatos no incluye los chunks.
3. Estrategias de indexación

Objetivo. Alinear índices con los filtros (find), ordenaciones (sort) y proyecciones más frecuentes.

  1. Inventaria índices: db.personas.getIndexes() o la salida de esta página.
  2. Para cada consulta crítica, ejecuta explain("executionStats") y verifica si aparece IXSCAN sobre el índice esperado o COLLSCAN.
  3. Reglas prácticas: igualdad antes que rango en índices compuestos; un índice que sirva a filtro + orden reduce costes de memoria en SORT; índices redundantes duplican coste de escritura.
  4. Texto completo o búsquedas semánticas en multimedia suelen requerir índice text o Atlas Search — documenta la alternativa elegida en tu análisis.
4. Costo de consulta — explain("executionStats")

Objetivo. Estimar el trabajo real del motor para una consulta concreta antes de escalar datos.

  1. Replica la consulta real: mismo filtro, orden, proyección y límite. Cambiar cualquiera puede cambiar el plan.
  2. Pide modo executionStats (no basta solo con queryPlanner si quieres números ejecutados en esta corrida).
  3. Lee en primer lugar: executionTimeMillis, nReturned, totalDocsExamined, totalKeysExamined. Idealmente totalDocsExamined cercano a nReturned cuando el filtro es selectivo.
  4. Inspecciona el árbol de etapas en winningPlan: busca COLLSCAN, IXSCAN, FETCH, SORT. Un SORT sin índice adecuado puede ser costoso en RAM.
  5. Repite tras crear o eliminar un índice y documenta la diferencia (antes/después). Eso es evidencia cuantitativa para tu memoria.
Las métricas de explain son para el estado actual del cluster y del tamaño de colección; extrapolar a millones de documentos requiere pruebas de carga o modelos más elaborados.

Proyección de almacenamiento

Fórmula orientativa: bytes nuevos ≈ documentos nuevos × avgObjSize (datos); los índices añaden overhead por campo indexado. En multimedia con GridFS, suma colecciones fs.files y fs.chunks más sus índices.

En producción sustituye el porcentaje por el ratio medido totalIndexSize / size tras establecer todos los índices definitivos.

Cómo crear estos gráficos para otros análisis

El patrón es siempre el mismo: (1) obtener datos estructurados → (2) transformarlos a arrays para el gráfico → (3) elegir el tipo de visualización según la pregunta estadística.

Paso 1 — Origen de los datos

  • API propia (como /api/metricas/resumen): encapsulas comandos MongoDB y devuelves JSON seguro para el navegador.
  • Agregaciones: por ejemplo histograma de edades con $bucket o serie temporal de altas con $group por día sobre creadoEn.
  • Exportación: CSV/JSON desde Compass o mongoexport y gráficos offline en Python (matplotlib) o Excel — la lógica de “labels + valores” es idéntica.

Paso 2 — Formato para Chart.js

Un gráfico minimal necesita:

{
  type: 'bar',   // o 'line', 'doughnut', 'scatter'...
  data: {
    labels: ['A', 'B', 'C'],
    datasets: [{ label: 'Serie 1', data: [12, 19, 3], backgroundColor: '#3ecf8e' }]
  },
  options: { responsive: true, plugins: { legend: { display: true } } }
}

Tu código debe mapear la respuesta del servidor a labels y data. Ejemplo: para índices, labels = Object.keys(indexSizes) y data = Object.values(indexSizes).

Paso 3 — Elegir tipo de gráfico

  • Barras: comparar categorías (tamaño por índice, BSON por documento, rangos de tamaño de archivo).
  • Líneas: evolución temporal (consultas/día, tamaño acumulado del bucket multimedia).
  • Circular / anillo: partes de un total (datos vs índices; tipos MIME en metadatos).
  • Scatter: relación entre dos variables continuas (duración de consulta vs documentos examinados en benchmarks repetidos).

Buenas prácticas

  • Destruye la instancia anterior con chart.destroy() antes de crear otra sobre el mismo <canvas> (esta página lo hace).
  • Ajusta tooltips para mostrar unidades humanas (bytes, ms) como en fmtBytes.
  • Si los datos son muy desbalanceados, usa escala logarítmica o normalización (como en el gráfico de explain) y documenta la transformación en el informe.
