Glosario rápido

HTTPProtocolo de comunicación entre cliente (navegador, Postman, esta página) y servidor.
PeticiónMensaje que envía el cliente: método + URI + cabeceras y a veces cuerpo.
URIIdentificador del recurso; en APIs REST suele ser el path que empieza en /, por ejemplo /api/personas. No incluye por sí solo el servidor ni el puerto.
RespuestaMensaje que devuelve el servidor: código de estado (número), cabeceras y cuerpo (HTML, JSON…).
MétodoVerbo que indica la intención: GET leer, POST crear, DELETE borrar, etc.
EndpointRuta concreta expuesta por la API (recurso o acción). Aquí: colección /api/personas o un documento /api/personas/<id>. Es el “punto de entrada” lógico del backend.
CabecerasMetadatos de la petición o respuesta. Ej.: Content-Type: application/json dice que el cuerpo es JSON.
Cuerpo (body)Datos que viajan en la petición (típico en POST/PUT). En JSON va entre llaves.
Código de estadoNúmero de tres dígitos: 2xx éxito, 4xx error del cliente, 5xx error del servidor.

URI, endpoint y URL: qué es cada uno

URI (path)
Camino del recurso desde la raíz del sitio. Ejemplos: /api/personas (colección) o /api/personas/673a2c9e1f... (un documento por _id). Es lo que muchos llaman “la ruta” o parte del endpoint.
URL completa
Incluye cómo y dónde contactar al servidor: esquema + host + puerto + path. Ejemplo: http://localhost:3000/api/personas. Postman suele pedir este formato en la barra principal.
Endpoint (uso práctico)
Es la “entrada” de la API que combinas con el método HTTP: misma familia de rutas (/api/personas, etc.). No es un cuarto formato distinto: es el contrato servidor/cliente sobre qué path usar para cada operación.
Campo del simulador (input URL)
Puedes pegar cualquiera de los dos: la URL completa o solo la URI que empiece por /. Si empieza por /, esta página antepone automáticamente el origen de esta pestaña (window.location.origin), así evitas errores de puerto si abriste el simulador desde tu mismo npm start.

Resumen: endpoint = qué recurso atacas; URI = path (/api/...); URL = protocolo + servidor + path. El campo del simulador acepta URL completa o solo path.

Métodos HTTP más usados

GETObtener datos (listar o leer). No debe cambiar el servidor. Sin cuerpo habitualmente.
POSTCrear un recurso o enviar datos. Suele llevar body (ej. JSON).
PUTReemplazar por completo un recurso existente (REST).
PATCHActualizar solo parte de un recurso.
DELETEEliminar un recurso. A veces el id va en la URL.

Este proyecto expone GET, POST y DELETE en /api/personas.

¿Dónde va cada cosa (como en Postman)?
  • Método: selector a la izquierda (GET, POST…).
  • URL: campo grande central — ahí va la dirección completa o la ruta del endpoint.
  • Body: pestaña “Body” en Postman; aquí el área de texto JSON solo aplica a métodos que envían datos.
  • Headers: esta demo añade automáticamente Content-Type: application/json cuando envías JSON en POST.
  • Send: envía la petición y debajo verás el código de estado y el cuerpo de respuesta.

Ejemplos para copiar y pegar

Pulsa Copiar, pega en el campo URL (o en el cuerpo JSON si aplica), elige el método y envía. Si tu app corre en otro puerto, usa la fila que empieza por /api/ estando en esta misma pestaña.

GET — Listar personas (GET /api/personas)

Sin cuerpo JSON. Dos formatos válidos en el campo URL:

http://localhost:3000/api/personas
/api/personas

POST — Crear persona (POST /api/personas)

URL + cuerpo JSON (fecha ISO YYYY-MM-DD):

http://localhost:3000/api/personas
/api/personas
{
  "nombre": "Alumno demo",
  "fechaNacimiento": "2002-06-15"
}

DELETE — Borrar una persona por _id

Sustituye REEMPLAZA_CON_ID_DEL_GET por el valor de _id que devolvió el GET (cadena hexadecimal). 

http://localhost:3000/api/personas/REEMPLAZA_CON_ID_DEL_GET
/api/personas/REEMPLAZA_CON_ID_DEL_GET
